# painpoint-lens — 把喵喵書評「轉版」成 painpoint.tw 拆書文章

## 這支 skill 在做什麼(以及為什麼不能複製貼上)

山姆的書評主場是 meowbooks.tw。painpoint.tw 是《痛點》的官網,要的是「痛點視角」的獨家內容當 SEO 引擎 + 漏斗入口。

**鐵則:同一篇書評,meowbooks 和 painpoint 不能放一模一樣的。** 兩個網域放重複內容 → Google 擇一顯示、互相搶排名,兩邊都掉。所以 painpoint 版**必須是換角度重組的獨家文章**,不是換排版的同一篇。

差異化的方法只有一個、而且很明確:**meowbooks 那篇是「書評」(誠實判決、誰該買誰別買、給星等);painpoint 這篇是「拆書」——拿 P.A.I.N. 這把尺,一格一格去戳這本書。** 同一本書、同一批事實,但骨架、開場、重點、標題全不同。

## 輸入

- 一篇**已定稿的喵喵書評**(山姆貼進來,或我剛在 Claude Code 寫完的那篇),外加書名與作者。
- 沒有書評也行:直接以這本書的內容 + P.A.I.N. 重寫一篇拆書(角度照樣是「拿尺戳書」)。

## 產出(兩個檔 + 一筆 registry)

1. `site/lens/{slug}.php` — 頁面檔。`{slug}` = 英文書名 kebab-case(例:Business Model Generation → `business-model-generation`、Competitive Strategy → `competitive-strategy`)。
2. `site/lens/_content/{slug}.html` — 正文(純 HTML,語意標籤,放進 `.reading` 裡)。
3. **在 `site/lens/_articles.php` 最上面加一筆**(最新在前):`slug`、`kicker`、`title`、`blurb`、`category`、`date`。**列表頁 `index.php` 與 `sitemap.php` 都讀這份清單自動長——不要再手改 index.php 的卡片或 sitemap 的網址,nav 也不必動。**

   ⚠️ **`blurb` 是「卡片一句話介紹」,硬上限 ≤ 160 字(漢字),絕不可貼整篇正文。** 這是 registry 欄位、會直接印在 `/lens` 卡片上,**它不是正文的摘要、更不是正文本身**。寫法照站上既有短 blurb 的節奏:〔書的核心一句 + 一個畫面〕→「用痛點的眼光重讀:」→〔戳向讀者點子的一句〕,2～3 句、收乾淨。**正文歸 `_content/{slug}.html`,blurb 歸 blurb,兩者不可互相照搬**(否則卡片變成一面牆,且 listing 與內文重複內容、SEO 兩敗)。加完一筆,順手 `php -r` 量一下這筆 blurb 的 `mb_strlen`,超過 160 就砍。

### `category` 的主分類:一律從受控清單挑(別亂開新詞)

`category` 寫成「主分類／次分類」(全形 ／;單一分類不必加 ／)。**/lens 的分類頁籤、以及卡片右上角的識別,都只看主分類(／前半);次分類忽略、一篇只歸一類。**

⚠️ **頁籤是「看 registry 有哪些主分類」即時長出來的——新主分類會自動長一個新頁籤(已實測)。** 好處是零維護;壞處是若每本書用略不同的詞,頁籤會爆量、變亂(違反設計憲法第 1 條)。所以:

**主分類一律從這份固定清單挑(= 喵喵書評的主分類清單,兩站一致):**
**策略 / 創新 / 行銷 / 管理 / 領導 / 行為 / 創業 / 理財 / 生產力 / 敘事 / 其他**

- 遇到清單沒有的主題,**對應到最接近的既有分類,不要發明新詞**:投資 / 財務 → 理財;組織 → 管理;決策 → 行為;溝通 / 談判 → 行銷或領導。
- 真要開一個新主分類(某主題累積夠多、值得獨立成一格),才在清單加一個,並意識到它會自動多一個頁籤。

> 🏷 **理財類「自動帶 YMYL 投資免責」(2026-06-25 起,引擎自動、零逐篇手工):** 凡主分類標成「理財」的拆書,`partials/reading.php` 會在文末**自動**補一句 YMYL 邊界(「這是讀書拆解與個人觀點、不是投資建議;作者是書評人、不是理財顧問…」)。所以:(1)**理財類別在 `_content` 裡手寫投資免責**——會變成兩句重複;(2)把投資/財務書的主分類**正確標成「理財」**(投資/財務 → 理財,見上),YMYL 就自動到位;非理財篇(含像《魔球》這種沾統計但非理財的)一律不帶。同理,所有 /lens 文章的「這篇怎麼拆」署名/方法框也由 reading.php 自動長,不必逐篇寫。

## ⚠️ 動筆前第一步:適配檢查 —— P.A.I.N. 是鏡頭,不是模具

**最重要的鐵則:不是每本書都塞得進 P→A→I→N 四格。硬塞會出兩種爛文章——① 邏輯不通(這本書根本沒有「在痛的客戶」,卻硬寫「誰在痛」);② 一眼看穿是套模板(整站每篇都長一樣的四個標題 = AI 量產味 + SEO 主題深度扣分)。所以動筆前先判斷這本書屬於哪一檔,選對模式再寫。**

判斷只看一刀:**這本書到底在不在講「幫某個『客戶』解決某個『痛』」?**
- 越是講「怎麼做生意、服務客戶」(創業 / 行銷 / 商業模式 / 策略)→ 越貼 P.A.I.N.。
- 越是講「改變你自己 / 大腦怎麼想 / 怎麼管錢 / 怎麼管組織」→ 越是硬塞。

依適配度選三種模式:

**🟢 模式 A — 完整四格(書天生在講客戶與痛)。** 例:《獲利世代》《精實創業》《什麼才是經營最難的事》《瘋潮行銷》《競爭策略》《藍海策略》。走下面〈角度配方〉的完整 P→A→I→N 四題。

**🟡 模式 B — 鬆綁,只深入這本書真正在講的一兩個字母。** 例:《定位》《創新的兩難》《先問為什麼》《從A到A+》《第五項修練》。**不硬湊滿四格**:挑這本書本來就強的那一兩格深寫,其餘輕輕帶過,並誠實說「這本書強在 X,Y 它沒幫你」。標題可以只出現一兩個字母,或不用字母當標題。

**🔴 模式 C — 不要用四格骨架,改用《痛點》的「世界觀」另長一個角度。** 例:《原子習慣》《少,但是更好》《七個習慣》《思考,快與慢》《推力》《漫步華爾街》《富爸爸》《被討厭的勇氣》——這些書裡**根本沒有「在痛的客戶」**,套四格一定假。改接《痛點》更底層的那把尺(是世界觀,不是四個字母):止痛藥不是維他命、痛是算得出來的代價、先確認有人要再動手、別被漂亮劇本騙、把代價換算成能心算的數字。用這個世界觀去跟書對話。
- 例:《原子習慣》→〈用痛點的眼光看《原子習慣》:它在教你對自己做需求驗證〉(把「先確認有人要」翻成「先確認這習慣真的解你的痛,不是別人說該養」)。
- 例:《漫步華爾街》→〈它教你的不是怎麼賺,是怎麼『不被故事騙』〉(接「別被漂亮劇本綁架」那條線)。
這一檔的文章**結構自己長**(隨書而定),不照〈角度配方〉那四題;但聲音、紀律、內部連結、導流、收尾全照舊。

> **共同鐵則:寧可這本書只接得上痛點世界觀的「一個點」、把那個點寫深寫真,也不要硬把四格填滿、填出讀者一眼識破的假。品質與真誠 > 結構統一。** 而且讓結構隨書變化,正好使整個 `/lens` 專區不會讀起來像同一套量產——這對「不像 AI 寫的」和 SEO 主題深度都是加分。
>
> 拿不準是 A 還是 C 的書(B 那一檔),預設**往保守走**:寧可用模式 B/C 鬆一點,也不要硬上模式 A。

## 角度配方(🟢 模式 A 專用):用 P.A.I.N. 四問當骨架

模式 A 的文章骨架走這條線(但**每篇換開場、換打比方、換切入**,別變成同一套模板;模式 B 只取其中一兩格,模式 C 不用這套):

1. **開場(2～4 段):** 先公道地講這本書最厲害的地方(它解了什麼、為什麼值得讀),再轉折出「但它沒逼你回答的那一格」——也就是痛點之尺要補的刀。開場錨在這本書真實的內容或一個具體場景,不要套用萬用句。**🚫 開場第一句禁用兩個「換書名即可複用」的起手式——①「先給／把該給《X》的公道／給足」 ②「《痛點》通篇／從頭到尾在練一個很不浪漫的動作」**(實測各自成簇複用,《逆思維》與《致富心態》第一句近乎逐字 = 自製重複內容、踩遮書名紅線);第一句必須含一個只屬於這本書的具體錨點,任兩篇開場第一句不得近似。
2. **一個貫穿的【打比方】:** 用一個生活化小場景把這本書的核心招式講白(框「(這是打比方……」+ 邀請代入,括號內不夾論點)。**框外那句『點破這是比方』每篇換字面、別都用「這是打比方」——在「這只是個比喻/這只是個例子/這是個比喻/打個比方而已/舉個例子/比如說/(偶爾)打比方」之間輪替、整站平均(全站千篇一律就是 AI 指紋);括號內的邀請代入句也每篇換字面——嚴禁全站重複「你身邊大概也有一個這樣的人」這款句模(換對象、換處境寫)。**
3. **P → A → I → N 當「暗線」,不要當 h2 字面:** 拿這本書的某個概念/章節/工具,逐一對照四問——
   - **P 誰在痛:** 這本書怎麼幫(或沒幫)你具體點名那群人?痛點之尺補:找得到幾個真人去問。
   - **A 痛多深:** 這本書有沒有逼你看「痛到付錢」?補:看他已經為這痛花了什麼。
   - **I 你怎麼介入:** 這本書在「怎麼做」上強在哪?補:為什麼是你。
   - **N 數字:** 這本書對「怎麼賺」誠不誠實?補:進來的錢 cover 不 cover 出去的。
   (四題裡哪一題這本書本來就很強,就大方說它強——拆書是用尺量,不是硬挑刺。)
   > 🚫 **【硬規定】四個 h2 嚴禁逐字掛「P　誰在痛／A　痛多深／I　你怎麼介入／N　數字」字母模板**(實測 7 篇模式 A 全這樣掛 → 同框互看一眼是套版,《競爭策略》與《藍海策略》的 A／N 格結論甚至可互換)。P.A.I.N. 只當幕後暗線,**h2 一律用「這本書自己的語言」重寫**——學《什麼才是經營最難的事》那篇把判斷寫進標題(例:「帶人那一格寫得滿,攻誰那一格幾乎空著」)。而且**四格深淺要隨書不同**:挑這本最強／最弱的一兩格寫深,其餘帶過,別四格等量填空。
4. **邊界(別固定成一個 h2):** 把「它幫你到哪、幫不到哪」**優先融進內文最後一個論證段**;真要立 h2,**字面每篇換寫、嚴禁逐字「一句話的邊界」**(實測 10 篇逐字撞),且**邊界句不得用「它幫你 X、卻不 Y」反差骨架收**(與〈收尾〉第 0 步的反差句上限合併計數)。
5. **一個今晚／這禮拜能做的具體動作(收尾意圖,非固定 h2):** 給讀者一個帶得走的動作(接到 N、接到「你」身上)。**但 h2 字面每篇必須換寫——嚴禁全站逐字「今晚就能做的一件事」**(實測 31/31 逐字撞);**且 CTA 段禁止再用「找 3～5 個真人＋問他已經為這件事花了多少錢＋導 /tool」這套共用句模**,動作要從這本書的內容長出來(換成這本書專屬的一個檢查、一個今晚能寫下的判斷)。

## 聲音與紀律(沿用 meowbooks-book-review 的尺)

- 聲音 DNA:短句重擊、數字變感受、破折號補刀、【打比方】框假想小人物、不說教金句、我講故事你收教訓。**全形標點。**
- **反 AI 味**是最高鐵則(逐條套 `skills/meowbooks-book-review/SKILL.md` 的〈反 AI 味自檢清單〉)。**特別盯 M 條庫存梗:「止痛藥 vs 維他命」這組詞整站省用、絕不照搬標準句式**(實測《藍海策略》逐字照搬、全站維他命 9 篇／止痛藥 6 篇),要講那個分別請用這本書自己長出來的話(模式 C 用《痛點》世界觀時最容易吐這個梗)。
- 但**拿掉書評專屬的東西**:① 不放文末「評分／分類／※註」那三行(那是 meowbooks 後台用的);② 不走「誰該買誰別買」的判決收尾;③ 第一人稱維持「讀過整面書牆的驗貨讀者」,喵喵創業數字預設不掏。
- **不要照搬 meowbooks 那篇的句子。** 開場第一句、打比方、結尾,尤其不能跟那篇撞——這篇若跟那篇太像,等於自製重複內容,整件事就白做了。

## 內部連結與導流(painpoint 版才有的任務)

**先懂新現實:/lens 的「文末頁尾」現在由 `site/partials/reading.php` 自動合成,你不用、也不該自己設計頁尾(2026-06-27 起)。** 引擎會自動長出:接著讀(相關拆書 grid)→ 探索行(同主題頁/跨主題書架,依 slug 輪換)→「訂閱拆書電子報」捕捉框(或在創業/行銷/創新/策略類**偶爾**換成 `/tool` 工具框)→ 一行「《痛點》電子書」書況註腳 → 上一篇/下一篇。**也就是說:買書、訂閱、工具的「頁尾」CTA 引擎都已經幫你放好、且已經適度化了**——你產新拆書唯一要顧的導流,只剩「**正文(`_content` html)裡的內鏈節制**」。

**CTA 北極星(山姆 2026-06-27,凌駕一切導購):CTA 的目標是把讀者留在站上——多讀一篇拆書、停留更久,不是把這篇變成賣書頁或工具導購頁。** 拆書最大的價值是內容本身;**過量內鏈 CTA(不管導工具還導書)會毀掉內容價值感、把人推離站外**。每多讀一篇,買書的種子已自然埋下,不必靠 CTA 硬推。

**正文 `_content/{slug}.html` 內鏈鐵則(新拆書最容易犯、最該守):**
- **整篇內文最多 1 個 `/tool` 連結,能 0 就 0。** 只有當這本書讀完「自然會讓人想去驗一個自己的點子」(創業/行銷/創新/策略/商業模式那類),且那句放在文章**後三分之一、靠近收尾**時才放;判準=把連結拿掉後那句仍完整通順(連結是順手一提、不是句子的目的)。**改變你自己/大腦/理財/習慣/管理/領導那類書(行為、理財、生產力、領導、敘事、管理)預設 0 個**——讀者讀完不一定有新點子要驗,硬接像業配。
- **結尾段一律不准放 `/tool` 連結**(消滅「最後一段=導工具」的收尾指紋);結尾用畫面/動作收。
- **絕不把整句做成連結**(像「花五分鐘做一次線上痛點體檢」「先拿 P.A.I.N. 體檢工具把這個點子跑一遍」)——錨文字只套在乾淨的名詞片語(例「P.A.I.N. 體檢工具」)上;不然就不放。
- **`/book` 內文連結:最多 1 個、多數篇 0 個。** 只在正文第一次自然提到《痛點》這把尺、且那句正好在說「這個判斷出自哪/書裡有段方法直接回應本篇」時,才把《痛點》三字做成連結;純為推書的 /book 一律不放。同一段落絕不並排 /tool 與 /book。書況一律「電子書已上市」,絕不寫即將上市/正在出版/實體書/留 email。
- `/read/pain-framework`(框架總覽)視情況連一次即可,別堆。

**`.php` 那層你只要顧(頁尾交給引擎、別自己組):**
- `$reading['funnel']`:仍寫 `title` + 一句 `body`(只有當引擎把這本書分到「偶爾工具框」時會沿用 title/body 首句,所以 **body 不要提買書**);**不要再設 `ghost`**(頁尾引擎不吃它了)。
- `$reading['prev']` / `['next']` 設 `null` 即可(reading.php 會用 `_articles.php` 全站順序自動接回線性上下篇)。
- `$reading['book']` 照填被拆的書名+作者(供 `pain_lens_article()` 產 Article + about/isBasedOn 結構化資料)。

## 檔案範本

`site/lens/{slug}.php`(照 `site/lens/business-model-generation.php` 改):設 `$page`(title/desc/path/type=article)、**`$reading`**(kicker、heading、published、updated、content 路徑、book、back=`['/lens','← 全部拆書']`、root=`['痛點之尺','/lens']`、funnel、prev/next)、`$page['schema'] = [pain_lens_article($cfg,$page,$reading), pain_person(...), pain_breadcrumb(...)]`,最後 require head / header / reading / footer。**⚠️ 長文外框 `partials/reading.php` 吃的變數叫 `$reading`,不是 `$lens`——名字寫錯會整段正文渲染不出來、外框退回「免費試讀」預設。**
> 🏷 **`<title>` 不用手寫公式:** `partials/head.php` 會對 `/lens/` 文章自動把 `<title>` 組成「`{$reading['heading']}｜《主書名》拆書・痛點 P.A.I.N.`」(書名自動切掉副標/英文名)。所以 `$page['title']` 寫什麼都會被蓋掉(留著當保險即可、別再費心寫「用痛點之尺重讀《X》」那種公式)。**你真正要顧的是 `heading` 寫成一句去公式、只屬於這本書的好文案**——`<title>` 跟 `<h1>` 都吃它,它好,兩個都好。`desc` 仍要每篇手寫(那是 meta description)。

`site/lens/_content/{slug}.html`:`<h1>` 標題 + 多個 `<h2 id="...">` + `<p>` + `<strong>` + `<blockquote><p>…</p></blockquote>`。純語意 HTML,**零 Markdown 符號**。**⚠️ `<h1>` 嚴禁套「用痛點的眼光看／痛點之尺,重讀／拆《書名》」公式**(實測約 28/31 都這樣寫,SERP 一列出來像同一台機器產的系列頁)——直接複用或對齊 `_articles.php` 那條具體的卡片 `title` 寫法(例:「最危險的黑天鵝,住在你自己的試算表裡」),把書名移到副標或內文首段。卡片標題寫得活、H1 就不該退回模板。
> 📏 **【硬上限・機械閘門】標題只放「一句鉤子」:卡片 `title`／頁面 `heading`／內文 `<h1>` 三處同一句、各 ≤ 32 字(漢字),且不准出現全形分號「；」。** ——2026-06-28 那批漂移,把本該放 blurb 的「作者說 X;你的盲點 Y」整段壓進標題(最長 125 字、篇篇用「；」接兩段),就是栽在這裡沒守。要兩個節拍用「,」或破折號;第二段、書名、作者的論點一律移回 `blurb`,別塞進標題。**收尾必跑 `python3 tools/count-title.py {slug}`,超長或含「；」CI 直接擋部署。**(範例的短促手感:「最危險的黑天鵝,住在你自己的試算表裡」「畫得出商業模式,不等於有人要買」。)
> 🔁 **標題別套同一句口號當鉤子:**「一定有人要」「有沒有人要」是 P.A.I.N. 的核心概念,放 `blurb`／內文無妨,但**別篇篇拿它當標題**——實測曾有 21 篇標題重複這句,列表跳針、SERP 像同一台機器產的系列頁(踩 E-E-A-T「規模化內容濫製」)。標題改用「這本書自己的畫面」講同一件事(隨機號碼喊的價／結論席／重讀上千遍的篤定…),讓整站口號零重複、每張卡片各長各的臉。

## 交稿前內容雙閘(SEO 承重,每篇必過)

這兩關不是文采要求,是 SEO 護城河(對齊 Google 的「helpful content / 經驗 E / 規模化內容濫製」判準),每篇定稿前都要過:

1. **第一手經驗 beat(硬閘,至少一處,寫不出 = 未定稿、不准輸出):** 文章裡要有一段「驗貨讀書人」**親身**的具體痕跡——而且必須是「讀過整面書牆才下得出的具體判斷」:把這本書放回它的書架座標(它比同類強／弱在哪、哪段是老調)、或這類書我讀過幾本通常解不了什麼,而這本書剛好戳中。**🚫 禁用充數寫法:免責聲明(「我不替它打分數」)、泛泛事實(「快六十年還沒過期」)、二手學術轉述(「後來有學者重算」)都不算 beat。** 這是 AI 摘要永遠抄不來、Google 2024 起最看重的「經驗(Experience)」訊號——**實測 31 篇只有 8 篇真的有、其餘 23 篇缺,是全站最大空洞**。(註:喵喵創業數字仍照〈聲音與紀律〉預設不掏,經驗 beat 用讀者眼光那條線即可。)
2. **反模板自檢(遮書名測試):** 寫完把全文的書名遮掉——**若超過 2 段拿掉書名還讀得通,就是模板化薄文,重寫。** Google 判「規模化內容濫製」看的是每篇有沒有真價值,不是看你寫了幾篇、或共用同一個頁面模板。會出事的永遠是 `_content/{slug}.html` 內文空,不是頁面外框或四個固定標題。
3. **跨篇結尾／開場硬閘(每篇必過):** 結尾 h2、邊界 h2、開場第一句、CTA 段**都不得與既有任一篇同字面或近似**——逐字「今晚就能做的一件事」已 31/31 撞、「一句話的邊界」10 篇撞,新文一律不准再撞;CTA 禁止再用「找 3～5 個真人＋問花了多少錢＋導 /tool」共用句模。寫完開 `site/lens/_content/` 抽兩三篇舊文,把你的開場第一句、四個 h2、結尾句並排比一遍,像 = 重寫。
   - **換皮指紋黑名單(被禁的舊指紋換字面、結構照舊重生的;`count-contrast.py` 會自動掃前三個,命中就改寫):** ①「我得替它把話講白／我得說一個邊界／我得誠實…」**「我得…」進邊界鉸鏈**(至少半數段落別這樣進邊界,改成直接給反例或先講邊界);②「這是打比方。……你身邊大概也有……的人」**打比方免責語固定模板**——**(a)『這是打比方』這個點破標籤全站篇篇一樣本身就是指紋**:點破語要在「這只是個比喻/這只是個例子/這是個比喻/打個比方而已/舉個例子/比如說/(偶爾)打比方」間輪替、整站平均;(b)後半每篇重新長字面,或半數篇不用【】改用書內實例;③ 判決框「(有)兩種人……第一種……第二種……」(painpoint 本就拿掉判決結構,更不准長出這個);④ painpoint **自製萬用開場**「先給該給的公道」「《痛點》通篇在練一個很不浪漫的動作」一律不准再用,開場錨在這本書專屬的東西。

## 規模化路線圖:/lens 不管長到幾篇都要守的「設計憲法」+ 應變階段

**設計憲法(山姆定的 5 條,每篇文章、每次改版都要守,不可為了規模犧牲):**
1. **讀者體驗**:不管 10 / 100 / 200 / 500 篇,頁面都要簡單、乾淨、好讀好找。
2. **SEO / E-E-A-T**:全程符合 Google 搜尋與 E-E-A-T(親身經驗 / 專業 / 權威 / 可信)。
3. **內部連結(節制)**:頁尾 CTA(訂閱電子報/偶爾工具/書況註腳/接著讀)由 `reading.php` 自動合成,別自己組。**正文內鏈硬上限:整篇最多 1 個 `/tool`(能 0 就 0、結尾段不准放、改變你自己那類書預設 0)、最多 1 個 `/book`(多數篇 0)。** 拆書是內容價值、不是篇篇導購買書/推工具(過量會傷 UX);只在這篇角度真的自然、合適時才連,絕不把整句做成連結。書況一律「電子書已上市」,絕不寫即將上市/正在出版/實體書/留 email 通知。
4. **高品質且不重複**:內容夠硬,且**絕不與 meowbooks 的書評重複**(見頂部重複內容鐵則)。
5. **不硬塞 P.A.I.N.**:讀者不能覺得是硬套框架(見〈適配檢查〉)。

**依文章數應變(到門檻才做,別提早過度建設;每階段都疊在 `_articles.php` 這份清單之上。門檻已依 2026 Google 現實校正——分頁/分類頁過去寫太早):**

- **現在～數十篇:單頁清單,已就緒。** 資料驅動(`_articles.php` → `index.php` 迴圈)、最新在前、卡片是伺服器端 `<a>`(JS 頁籤只切 `hidden`,SEO 安全)、文末「相關拆書」已由 `reading.php` 自動長(同主分類優先、不足補最新、最多 4 篇,描述性錨文字)。**先別加分頁、別蓋分類頁**——分類太薄、且爬取預算在數千頁內都不是問題(每篇靠 sitemap 直接被收錄)。
- **約 40 篇:分類主題頁(為了賺 topical authority,不是為了爬取)。** 只給**滿 5～8 篇以上、且寫得出 150～300 字山姆親寫導言**的主分類做 `/lens/c/{分類}`(自我 canonical、進 sitemap、`CollectionPage`+`ItemList`+`BreadcrumbList`);不夠厚的分類維持純前端頁籤、不進 sitemap。**硬閘:湊不夠篇數又沒獨特導言就別開**,否則就是 Google 會降權的薄頁 / doorway(違反原則 5)。把現有 JS 頁籤升級成指向這些真實網址(漸進增強:連結能走、JS 攔截做即時篩選)。列表 / 分類頁的 `ItemList`(`pain_lens_itemlist()` 已備)可擴成 `CollectionPage`。
- **約 100 篇:站內搜尋 + 第二條聚類軸。** 輕量前端搜尋(`_articles.php` 生 JSON 索引;**純給使用者**——SearchAction / Sitelinks Search Box 已於 2024-11 廢除,不會在 SERP 長出搜尋框)。相關模組再加一條軸(例:同作者 / 同領域互鏈)。作者權威頁做滿、每篇署名都指向同一個 Person `@id`。
- **約數百篇,或單頁 CWV 真的變差時:才減輕列表頁重量。** 由實測 Core Web Vitals(LCP / DOM 大小)觸發,**不是由篇數**。分頁只是其中一種解(也可:伺服器端先渲染最新 30～50、其餘走分類頁 + 扁平總表)。若分頁:真實可爬 `?p=N`、每頁**自我 canonical**、真 `<a>` 上下頁 + 頁碼、每頁連回 `/lens`、每個 `?p=n` 進 sitemap。**絕不**用 rel=next/prev(已廢、無作用)、**絕不**把第 2 頁 canonical 指回第 1 頁或 view-all(會把深頁踢出索引)、**絕不**用 `#` 片段或 JS 無限捲動(Google 不會捲 / 點,內容變隱形)。

**各階段共同鐵則:** 加任何結構都不能犧牲那 5 條;**E-E-A-T 最大殺手是薄內容,不是頁數或共用模板**(見〈交稿前內容雙閘〉)。**跨網域(meowbooks)風險其實低**:Google 沒有「重複內容懲罰」,「自相殘殺(cannibalization)」也被官方淡化——只要 `/lens` 維持拆書角度、不抄句子,就不會跟 meowbooks 那篇互打;**但千萬別在 `/lens` 與 meowbooks 之間設 cross-domain canonical**(那會真的讓其中一頁被踢出索引,正是要避免的結果)。技術上一律沿用既有 partials / CSS、走 `deploy.sh`、改 CSS 記得 bump `head.php` 的 `style.css?v=`。
